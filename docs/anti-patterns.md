# Anti-Patterns

Six approaches that were tried during the design of this pattern and deliberately rejected. Each is documented with the original idea, the failure mode that surfaced, and the replacement. This document exists so future contributors do not re-invent these rejections — and so users understand why the pattern looks the way it does.

The thread connecting all six: **adaptivity belongs in a single dedicated meta-agent, not in the orchestrator's rule engine, not in distributed coordination protocols, and not in N parallel role-players.**

---

## 1. Parallel agent army without a chain

**Idea.** Eight agents in parallel, each doing breadth research on a different angle, then a consolidator pulls them together. Maximum wall-clock efficiency.

**Failure mode.** No learning effect between agents. Each agent rediscovers the same basics (the prominent frameworks, the well-known tools) because none of them has the others' outputs. The consolidator receives eight redundant outputs and cannot calibrate between them — the parallel agents had no way to specialise away from each other. The orchestrator pays for redundant research and gets no compositional gain.

**What replaced it.** A sequential chain. Agent N reads agents 1 through N-1's outputs as context. The breadth agent goes wide; the methodology agent goes deep on the most promising candidates the breadth agent surfaced; the orchestration agent goes deeper on combinations the methodology agent flagged. Each agent's output is the next agent's calibrating input.

**When parallel is the right answer.** When the agents are doing fundamentally independent work — separate codebases to investigate, separate web searches whose results do not inform each other, separate eval suites. Not when they are doing different angles on the same research question.

---

## 2. Six separate persona agents (stakeholder theatre)

**Idea.** One agent per stakeholder — a "Compliance Officer Agent", a "GDPR Lawyer Agent", a "Pilot Customer Agent", a "Works Council Agent", an "Enterprise IT Agent", a "Competitor Agent". Each plays its role and produces stakeholder-specific concerns. Looks thorough.

**Failure mode.** The model is playing self-invented roles, and the seams show. The differences between the six outputs are marginal because they are all generated by the same underlying model, in the same context window of training, with the same broad understanding of "what compliance officers think". You get six monologues that share 60% of their content. Costs are six-fold; insight is not.

Worse: by spawning N agents per stakeholder, the most valuable insight — **conflicts between stakeholders** — is lost. Each agent sees only its own perspective. There is no in-context reasoner who notices that Compliance and Engineering would disagree on a specific decision.

**What replaced it.** A single Stakeholder-Sweep agent that holds all relevant perspectives in one context. It reasons about each stakeholder's concerns and, crucially, surfaces cross-stakeholder conflicts as a deliberate output section. One agent, six perspectives, one integrated analysis, conflicts visible.

**When N persona agents would be the right answer.** Honestly, almost never with current models. If perspectives are genuinely disjoint and require deep specialised research (different regulatory documents, different industry contexts), a single sweep with explicit stakeholder-by-stakeholder structure beats N theatre agents in our experience.

## 3. Divergence-then-convergence phase split

**Idea.** Run three "divergence" agents that go very wide (no constraint), then a synthesis step, then four "convergence" agents that go narrow on the synthesis. Sounds clean, sounds methodologically rigorous.

**Failure mode.** No measurable quality difference vs. a chain that simply puts the breadth agent first. A single agent can move from divergent to convergent reasoning on its own — that is what good prompting already produces. Imposing the phase split as architecture adds two extra agents (synthesis + structured second wave) without producing better output. It introduces ceremony that costs tokens and time.

**What replaced it.** Wildcard-Breadth as Agent 1 (intentionally divergent), focused research from Agent 2 onward (intentionally convergent), Gap-Hunter at the end as the meta-divergent check. The same effect with three fewer moving parts.

**When the explicit split is the right answer.** When divergence and convergence require fundamentally different tooling — e.g. divergence is web-search-heavy and convergence is internal-codebase-heavy. Two distinct tool stacks may justify two distinct phases. Not when both phases are doing language model reasoning over text.

---

## 4. Five-trigger rule engine for emergent spawning

**Idea.** The orchestrator carries five conditional rules — "if methodology agent finds X then spawn Y", "if stakeholder sweep mentions Z then spawn W". Detailed, declarative, auditable.

**Failure mode.** Too many guard-rails create bugs. Rules either over-trigger (spawning agents that produce noise) or under-trigger (missing the case the user actually cares about). More importantly, rules never cover all the emergent cases — and the cases they cover are usually the obvious ones, while the valuable emergent insight is in the cases nobody anticipated.

The orchestrator becomes complex. Debugging the rule interactions takes more time than the runs themselves. And every new project needs new rules, which negates the point of having a reusable pattern.

**What replaced it.** A dedicated Gap-Hunter meta-agent. The orchestrator stays "dumb" — it executes the chain. The Gap-Hunter is "smart" — it reads everything and decides what is missing. Adaptivity becomes a single-responsibility agent, not a rule engine. The Gap-Hunter is given a hard cap on emergent spawns (3) so it cannot run away.

This is **the central architectural insight of the pattern**. If you remember nothing else: adaptivity is a single agent's job, not the orchestrator's.

## 5. Full blackboard coordination (multi-file message passing)

**Idea.** A `questions-queue.md`, an `answers-log.md`, and a `brain.md` as a message-passing protocol. Multiple parallel terminals coordinate by reading and writing to these files. Sophisticated, distributed, scalable.

**Failure mode.** File lock contention, race conditions, debug nightmare. There is no native file-watcher in the runtime; agents poll, miss updates, overwrite each other. Designing the protocol takes more effort than the research itself. Recovery from a corrupted state file is manual archaeology.

For a single overnight batch, the infrastructure cost is wildly disproportionate to the benefit. Distributed coordination is the right answer for distributed systems running continuously, not for a sequential chain that runs once.

**What replaced it.** A single append-only `brain.md`. Sequential writes. Zero race conditions. Zero protocol design. The audit trail is the file itself.

**When blackboard would be the right answer.** Long-running multi-day systems with multiple persistent agents and asynchronous coordination requirements. Not for a research run that completes in a single overnight batch.

---

## 6. Integrating external multi-agent tools mid-run

**Idea.** Use a swarm-intelligence tool, a multi-agent simulation framework, or some other external orchestration product mid-research-run for "better" stakeholder simulation or perspective generation. Plug in MiroFish-class tools, AutoGen-class frameworks, or persona-simulation services.

**Failure mode.** Setup overhead dwarfs the gain. If the tool fails mid-run, the night is lost. Most such tools simulate opinions from seed text — structured hallucination, not external fact. The integration becomes the highest-risk component on the critical path of a long-running batch.

External multi-agent tools have legitimate uses (rapid prototyping, exploratory simulation, training data generation). They do not belong on the critical path of a one-shot overnight research batch where reliability dominates.

**What replaced it.** In-chain Stakeholder-Sweep with explicit perspective coverage. External tools remain available for isolated experimentation outside the research path.

**When external tools would be the right answer.** When the research has a long-term horizon and the external tool's setup cost amortises over many runs. Not when the goal is one decisive overnight batch.

---

## The thread

Every rejection above followed the same pattern: a more elaborate architecture promised better output, the elaboration introduced new failure modes that overwhelmed the gain, and a simpler architecture delivered the same insight at a fraction of the complexity.

This is not a coincidence. Multi-agent research systems that succeed in production share a few traits:

1. Adaptivity in a dedicated single-responsibility agent, not the orchestrator
2. Sequential composition for compounding insight, not parallel for wall-clock speed
3. Append-only shared state, not distributed coordination
4. Bounded recursion, not unbounded emergence
5. Honest filters that recommend against use when a simpler path fits

The Gap-Hunter pattern is the thinnest viable system that exhibits all five.
